# Execution Layer Design

**Status:** Implemented
**Last updated:** 2026-05-18

---

## Overview

The execution layer gives OpenClaw four distinct execution primitives for driving free-form pentesting. Scenarios and pre-baked manifests are removed — OpenClaw generates all inputs dynamically based on recon findings.

The threat model that defines scope:

> An attacker who can create workloads in a Kubernetes namespace — what can they do?

This covers the full post-admission attack surface while keeping execution deterministic and confined to the `chaosclaw` namespace.

---

## The Four Primitives

### Tier 1 — `chaosclaw verify run` (Manifest admission)

*Already implemented. Scenarios removed. OpenClaw generates manifests dynamically.*

Tests whether the cluster's admission control blocks or allows a given manifest.

```
--manifest <path>            Manifest to submit (generated by OpenClaw)
--expect <rejected|allowed>  Expected admission outcome
--context <name>             Kubernetes context
--namespace <name>           Target namespace (default: chaosclaw)
--output <path>              Write JSON result to file
--format <table|json>
```

**Verdict:** did admission match `--expect`? → `Pass` or `Fail`

---

### Tier 2 — `chaosclaw verify exec` (Runtime binary execution)

**Status: Implemented.**

Creates a pod, waits for it to reach Running, execs a command inside it, captures the result, cleans up. OpenClaw controls the pod entirely via the manifest — image, security context, service account, volume mounts.

```
--pod <path>                       Pod manifest (OpenClaw generates this)
--run "<command>"                  Shell command to exec inside the container
--container <name>                 Which container to exec into (default: first)
--expect <succeeded|failed|denied>
                                   succeeded = exit code 0
                                   failed    = non-zero exit code
                                   denied    = exec API call returned 403
--alert-source <tool>              Also poll: none (default), falco, tetragon, kubearmor
--observation-window <seconds>     Alert poll window (default: 10)
--pod-timeout <seconds>            Max wait for pod to reach Running (default: 60)
--exec-timeout <seconds>           Max time for the command to complete (default: 30)
--context <name>
--namespace <name>                 (default: chaosclaw)
--output <path>
--format <table|json>
--cleanup <always|on-success>
```

**Verdict:** did exec outcome match `--expect`? → `Pass` or `Fail`

**Evidence captures:** exit code, stdout (truncated at 4KB), stderr, whether the pod scheduled, cleanup status.

**What this can test (image-dependent):**
- Container escape via `nsenter --mount=/proc/1/ns/mnt`
- Host filesystem access via chroot or volume mounts
- Token theft — read `/var/run/secrets/kubernetes.io/serviceaccount/token`
- Sensitive file access — `/etc/shadow`, `/proc/1/root`
- Kernel module loading, `/proc` manipulation
- Any binary-level exploit that can be packaged into an image

### Tier 3 — `chaosclaw verify network` (Reachability from within a pod)

**Status: Implemented.**

Creates a pod, uses it to probe a target endpoint, records whether the connection succeeded or failed, cleans up.

```
--from <path>                      Pod manifest to use as the network source
--target <url|host:port>           What to probe
--protocol <http|https|tcp>        Default: inferred from target
--expect <reachable|unreachable>
--alert-source <tool>              Also poll: none (default), falco, tetragon, kubearmor
--observation-window <seconds>     Alert poll window (default: 10)
--connect-timeout <seconds>        TCP connect timeout (default: 5)
--pod-timeout <seconds>            Max wait for pod to reach Running (default: 60)
--context <name>
--namespace <name>                 (default: chaosclaw)
--output <path>
--format <table|json>
--cleanup <always|on-success>
```

**Verdict:** did reachability match `--expect`? → `Pass` or `Fail`

**Evidence captures:** protocol, HTTP status code (if HTTP), response time ms, error type if unreachable — timeout vs refused vs reset carry different meanings.

**Target examples OpenClaw generates from recon:**
```
http://169.254.169.254/latest/meta-data/    # cloud metadata service
https://kubernetes.default.svc              # Kubernetes API from inside pod
http://<pod-ip-in-other-namespace>:8080     # lateral movement
https://external.example.com               # egress / exfiltration path
http://<node-ip>:10250/pods               # kubelet API
http://<node-ip>:2379                     # etcd (if reachable)
```

### Tier 4 — `chaosclaw verify identity` (RBAC capability test)

**Status: Implemented.**

Tests what a specific service account is actually authorized to do against the Kubernetes API.

```
--as <sa-name>                     Service account to test (lives in --namespace)
--can <verb>                       e.g. get, list, create, delete, *, exec
--resource <resource>              e.g. secrets, pods, clusterrolebindings, pods/exec
--resource-namespace <ns>          Namespace to test the permission in
                                   (omit for cluster-scoped check)
--expect <allowed|denied>
--context <name>
--namespace <name>                 Namespace the SA lives in (default: chaosclaw)
--output <path>
--format <table|json>
```

**Verdict:** did authorization result match `--expect`? → `Pass` or `Fail`

**Evidence captures:** SA name, verb, resource, resource namespace, actual allowed/denied result.

**Examples OpenClaw generates from recon RBAC findings:**
```bash
# Can the default SA read secrets in kube-system?
chaosclaw verify identity --as default --can list --resource secrets \
  --resource-namespace kube-system --expect denied

# Can the chaosclaw-runner SA create pods outside its namespace?
chaosclaw verify identity --as chaosclaw-runner --can create --resource pods \
  --resource-namespace production --expect denied

# Can any SA exec into running pods?
chaosclaw verify identity --as default --can create --resource pods/exec --expect denied
```

### Tier 5 — `chaosclaw verify detect` (Runtime detection)

**Status: Implemented.**

Submits a pod, waits for it to reach Running, execs a threat command inside it, then polls the configured runtime security tool for a correlated alert. Reports whether the tool detected (or blocked) the action within the observation window.

```
--pod <path>                       Pod manifest (OpenClaw generates this)
--run "<command>"                  Threat command to exec inside the container
--container <name>                 Which container to exec into (default: first)
--expect <alert_fired|action_blocked|no_alert>
                                   alert_fired    = runtime tool emitted an alert
                                   action_blocked = runtime tool blocked the syscall
                                   no_alert       = observation window closed without a match
--alert-source <tool>              Alert source to poll: none (default). Future: falco, tetragon, kubearmor
--observation-window <seconds>     Seconds to wait for an alert after exec (default: 10)
--pod-timeout <seconds>            Max wait for pod to reach Running (default: 60)
--context <name>
--namespace <name>                 (default: chaosclaw)
--output <path>
--format <table|json>
--cleanup <always|on-success>
```

**Verdict:** did the observed detection outcome match `--expect`? → `Pass` or `Fail`

**Evidence captures:** alert source, rule name, pod name, triggered-at timestamp, raw alert payload.

**Alert sources:** currently only `none` (NullAlertSource) is implemented. Falco, Tetragon, and KubeArmor adapters are planned for Phase 2. With `--alert-source none`, the observed outcome is always `no_alert` — useful for dry-run pipeline testing.

**Example usage OpenClaw generates from recon findings:**
```bash
# Does Falco detect reading /etc/shadow?
chaosclaw verify detect \
  --pod ./shadow-reader.yaml \
  --run "cat /etc/shadow" \
  --expect alert_fired \
  --alert-source falco \
  --observation-window 15

# Does KubeArmor block nsenter attempts?
chaosclaw verify detect \
  --pod ./escape-probe.yaml \
  --run "nsenter --mount=/proc/1/ns/mnt ls /" \
  --expect action_blocked \
  --alert-source kubearmor
```

## Shared Evidence Schema

Every command produces the same top-level envelope:

| Field | Type | Description |
|---|---|---|
| `runId` | string | UUID |
| `type` | string | `manifest`, `exec`, `network`, `identity`, `detect` |
| `clusterContext` | string | Cluster that was tested |
| `namespace` | string | Namespace used |
| `startedAt` | string | ISO timestamp |
| `endedAt` | string | ISO timestamp |
| `status` | string | `Pass`, `Fail`, `Error`, `Skipped` |
| `expectedOutcome` | string | What was declared via `--expect` |
| `observedOutcome` | string | What actually happened |
| `cleanupStatus` | string | `complete`, `partial`, `failed` |
| `detail` | object | Type-specific fields (see below) |

Type-specific `detail` fields:

| Type | Fields |
|---|---|
| `manifest` | `admissionDecision`, `rejectionReason` |
| `exec` | `exitCode`, `stdout`, `stderr`, `podScheduled`, `command` |
| `network` | `protocol`, `httpStatus`, `responseTimeMs`, `errorType` |
| `identity` | `serviceAccount`, `verb`, `resource`, `resourceNamespace`, `allowed` |
| `detect` | `alertSource`, `ruleName`, `podName`, `triggeredAt`, `rawAlert` |

## Attack Chain Example

With all five primitives, OpenClaw can run a full attack chain from a single recon report:

```
recon rbac → finds non-built-in cluster-admin binding on SA "app-runner"

  → identity: can app-runner list secrets in kube-system?        (RBAC check)
  → run:      can a pod with app-runner SA be created?           (admission)
  → exec:     can it read /var/run/secrets/.../token?            (token theft)
  → identity: what can that token do?                            (escalation)
  → network:  can the pod reach the Kubernetes API directly?     (API access)
  → detect:   does Falco fire when the token is read?            (detection gap)
```

Each step's result informs whether to run the next.

## Known Gaps

### 1. Alert source adapters — configuration requirements
Falco, Tetragon, and KubeArmor adapters are implemented and work by polling pod logs. Each has a prerequisite:

- **Falco**: must be configured with `json_output: true` (Helm: `falco.jsonOutput: true`). Without JSON output, log lines are human-readable text and cannot be parsed.
- **Tetragon**: must be deployed with the `export-stdout` sidecar container (default in the Helm chart). TracingPolicies must be installed that cover the test actions.
- **KubeArmor**: KubeArmorPolicies must be installed targeting the test namespace. Without a matching policy, KubeArmor emits no MatchedPolicy events.

With `--alert-source none`, the observed outcome is always `no_alert`.

### 2. Multi-step chains with shared state
Each CLI call is stateless. "Steal token in step 1, use it to create a pod in kube-system in step 2" requires OpenClaw to chain calls, passing outputs from one as inputs to the next. The CLI does not express the chain itself — OpenClaw orchestrates it.

### 3. Mutation inspection
When a webhook mutates a submitted manifest, the CLI cannot capture the diff between the submitted spec and the admitted spec. Useful for testing security-relevant mutations (e.g., does the webhook strip privileged?).

### 4. Downstream effect observation
`run` can submit a DaemonSet manifest but cannot verify whether it actually scheduled on all nodes. There is no primitive for observing downstream scheduling effects.

### Out of scope entirely
- Control plane exploits (kube-apiserver CVEs, etcd direct access without a network path)
- Node-level persistence outside of Kubernetes primitives
- Timing / race condition exploits
- Interactive sessions
- Network-level attacks (ARP poisoning, MITM)

## Image Strategy (unresolved)

The `exec` primitive's power is bounded by what's available in the container image. OpenClaw controls the image via the pod manifest. This is an open design question:

**Option A: Purpose-built images per test category**
- `chaosclaw/network-probe` — curl, nc, wget, nslookup
- `chaosclaw/escape-probe` — nsenter, chroot, mount utilities
- `chaosclaw/api-probe` — kubectl, token tools

Pro: minimal, auditable, no unexpected tooling  
Con: OpenClaw must know which image to use per test type

**Option B: Single pentest image with all tools**
- One image: curl, nc, kubectl, nsenter, python, common exploit tools

Pro: simple, OpenClaw doesn't need to choose  
Con: large image, harder to audit, potential supply chain concern

**Option C: User-supplied image**
- OpenClaw generates the full pod manifest including image selection

Pro: maximum flexibility, no image maintenance burden on ChaosClaw  
Con: OpenClaw must have image knowledge; untrusted images in the test namespace

Decision deferred — resolve before implementing `exec` and `network`.
